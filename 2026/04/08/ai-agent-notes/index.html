AI Agent 入门：Anthropic 官方指南笔记

花时间翻了 Anthropic 那篇《Building Effective Agents》，顺手整理成学习笔记。

核心观点其实就一句：大多数成功的 Agent 实现，用的都是简单可组合的模式，而不是复杂的框架。

1. Agent 的核心工作原理

先区分两个概念：

  • Workflows：通过预定义代码路径来编排 LLM 和工具，步骤是固定的
  • Agents：LLM 动态决定执行流程，自己选择下一步用什么工具

说白了，Agent 就是带工具调用的 LLM + 环境反馈循环。它不是魔法，原理很朴素：做事 → 看结果 → 决定下一步 → 继续。

用 Anthropic 的话来说，Agent 靠的是四个成熟的能力：理解复杂输入、主动推理、使用工具、从错误中恢复。

2. 五种工作流模式

Anthropic 总结了五种常见模式，按复杂度从低到高：

Prompt Chaining（链式调用）

把任务拆成顺序步骤，每步的输出是下一步的输入。适合”先做A，再用A的结果做B”这种场景。

比如：写文案大纲 → 检查大纲是否达标 → 写正文。

Routing（路由）

先分类输入，再分发给专门的处理流程。核心价值是隔离不同类型的问题，避免互相干扰。

比如：客服问题分类 → 退款请求 / 技术支持 / 常规咨询走不同流程。

Parallelization（并行化）

两种形式：

  • Sectioning：拆成独立子任务同时跑，再合并结果
  • Voting：同一个任务跑多遍，用投票决定最终输出

适合需要多视角判断、或子任务相互独立的场景。比如代码安全审查，多个 prompt 各自挑刺，比一个 prompt 全面检查效果好得多。

Orchestrator-Workers（编排器-工人）

中心 LLM 动态拆解任务，分发给多个 Worker，再汇总结果。关键区别是子任务不是预定义的，而是由编排器根据输入决定的。

适合代码修改这类场景——每次要改哪些文件、改什么内容，输入不同结果就不同。

Evaluator-Optimizer（评估-优化）

一个 LLM 生成，另一个 LLM 评估，形成循环。适合有明确评估标准、迭代能带来可见提升的场景。

典型用法：翻译 + 评审 → 再翻译 → 再评审，直到达标。

3. 如何构建第一个 Agent

Anthropic 的建议很直接：从直接调用 LLM API 开始，很多模式用几行代码就能实现。

如果选框架（比如 LangGraph、LlamaIndex、CrewAI），确保你理解底层在做什么。框架会掩盖细节，出问题时排查困难。

实践建议：

  1. 先想清楚 Agent 要做什么、为什么需要它
  2. 从单一行为模式开始，不要一上来就多 Agent
  3. 明确输入输出，别让 Agent 自己猜
  4. 定义好停止条件（最大循环次数、超时等）

4. 工具使用的原则

Anthropic 提了一个有意思的概念：ACI（Agent-Computer Interface），对标传统的 HCI。

工具设计建议：

  • 参数命名要自解释，想象是在给新人写文档
  • 给足够的 token 空间让模型”思考”，别让它写到一半被卡住
  • 格式尽量贴近模型在互联网上见过的自然形式
  • 用真实输入测试，观察模型会犯什么错，然后改进

Anthropic 在开发 SWE-bench 编码 Agent 时，花在优化工具上的时间比优化整体 prompt 还多。

5. 记忆机制

Agent 需要从环境反馈中学习进度，这是”记忆”的核心：不是记住历史，而是实时获取当前状态。

实践中需要注意：

  • 每次工具调用后获取结果，作为下一步判断的依据
  • 设置检查点，必要时暂停等人类反馈
  • 控制 token 消耗，避免上下文膨胀

6. 测试驱动

Agent 的不确定性意味着测试格外重要：

  • 沙盒环境：先在隔离环境跑，观察行为
  • 可验证输出：比如代码任务有自动化测试，客服任务有明确解决标准
  • 边界条件：测试工具在边界情况下的表现（空输入、异常格式等）

Anthropic 的建议是：不要只测 Prompt，工具本身也需要用真实输入反复测试迭代。

7. 多 Agent 何时需要

当单 Agent 出现以下问题时，考虑拆分：

  • 任务类型差异大，一个 Prompt 难以兼顾
  • 某个子任务变得复杂且独立，值得单独管理
  • 需要并行处理不同类型的请求

但也要注意：多 Agent 带来的是协调成本和更多不确定性。能用单 Agent 解决就别拆。

8. 核心结论

Anthropic 的三个核心原则：

  1. 保持简单 —— 设计复杂不等于效果好
  2. 透明可见 —— 明确展示 Agent 的推理步骤
  3. ACI 优先 —— 像重视人机交互一样重视 Agent 与工具的接口

最后一句话值得记住：

构建正确的系统，而不是最复杂的系统。
