AI Agent 三叉戟：Claude Managed Agents、Hermes Agent、OpenClaw 技术横评

发表于2026-04-09|更新于2026-04-09
|总字数:3k|阅读时长:10分钟|浏览量:

AI Agent 江湖正在快速分化。Anthropic 官方下场托管，Nous Research 开源自进化引擎，OpenClaw 主打网关生态——三条路线各有取舍。本文从技术架构、记忆系统、技能系统、平台接入、安全模型六个维度横向对比，帮你在真实场景中做出选型判断。

TL;DR

  • Claude Managed Agents：Anthropic 官方托管的 meta-harness，解耦 brain/hands/session，容器按需启动，TTFT p50 降低约 60%
  • Hermes Agent：Nous Research 开源，9000+ 行单核心引擎驱动所有入口，自进化记忆 + 自动创建技能，支持 18+ 模型
  • OpenClaw：开源自托管网关，Gateway 中心化架构，接入 15+ 聊天平台，ClawHub 技能生态

一、技术架构：三条不同的抽象路径

Claude Managed Agents：OS 虚拟化思路

Managed Agents 的核心洞察来自一个古老的计算问题：如何为”尚未想到的程序”设计系统。操作系统当年用进程和文件抽象解决了这个问题——read() 写死了 1970 年代的磁盘和今天的 SSD 都能用。Anthropic 沿用同一思路，把 Agent 的三个组件全部虚拟化：

  • Session（日志）：append-only 事件流，通过 getEvents() 按位置切片获取，支持 rewind 和 lookahead，完全处于 Claude 上下文窗口之外
  • Harness（大脑）：调用 Claude 并路由工具调用的循环，Stateless，崩溃后用 wake(sessionId) + getSession(id) 重启恢复
  • Sandbox（手）：容器化的执行环境，通过 provision({resources}) 按需创建，通过 execute(name, input) → string 调用

最初它们是”宠物”——全部塞进同一个容器，容器挂了会话就丢了。后来全部改成” cattle”：独立替换、独立失败、独立恢复。

关键性能数字：p50 TTFT 下降约 60%，p95 下降超过 90%。原因很简单——不需要沙箱的任务不再等容器启动，推理从第一条消息开始就能进行。

Hermes Agent：单引擎多入口

Hermes 的架构与 Claude MA 截然相反——一个约 9200 行的 AIAgent 类（run_agent.py）驱动所有入口：CLI、消息网关、ACP 编辑器集成、批处理 runner、API Server。所有这些共用同一个 Provider 解析器、同一个工具分发逻辑、同一个记忆系统。

架构亮点：

  • Frozen Snapshot 模式：系统提示词在会话启动时冻结，之后不会中途修改——这保证了 LLM 前缀缓存的稳定，不会出现缓存失效导致的性能抖动
  • 可中断 API 调用：API 调用运行在后台线程，主线程监听 interrupt 事件；用户发 /stop 或新消息时立即放弃，无脏数据注入对话历史
  • 三种 API 模式动态解析：chat_completions（OpenAI 兼容）、codex_responses（Responses API）、anthropic_messages（原生）

OpenClaw：Gateway 中心化架构

OpenClaw 架构围绕一个长期运行的 Gateway 进程展开。Gateway 是唯一打开 WhatsApp（Baileys）、Telegram（grammY）等频道会话的进程，CLI、Web UI、macOS/iOS/Android 节点都通过 WebSocket（默认 ws://127.0.0.1:18789）连接 Gateway。

架构特点：

  • 设备级配对：新设备需要显式审批，Gateway 颁发设备 token 后续自动重连
  • 心跳系统：默认每 30 分钟唤醒一次，检查 HEARTBEAT.md，实现主动行为
  • TypeBox schema 定义的类型化 WebSocket API，请求/响应/事件三种消息类型

架构对比小结

维度 Claude MA Hermes OpenClaw
核心抽象 Session / Harness / Sandbox 单 AIAgent 类 Gateway + Runtime
架构哲学 OS 虚拟化（pet→cattle） 单引擎多入口 分层网关
Session 持久化 外部事件流 SQLite + FTS5 Markdown 文件
容器化 原生托管，按需 provision Docker / 多后端 Docker 可选
扩展方式 多 Brain 多 Hand 单引擎 + 平台适配器 Gateway 插件

二、记忆系统：持久化的三种思路

Claude Managed Agents：Memory Stores

处于 Research Preview 阶段，提供workspace 级别的持久记忆：

  • 每个 Session 最多 8 个 Memory Stores
  • Agent 在任务开始时自动读取，任务完成时自动写入学到的东西
  • 工具：memory_list、memory_search、memory_read、memory_write、memory_edit、memory_delete
  • 单条记忆上限 100KB（约 25K tokens）
  • 有不可变的版本历史，支持审计和回滚
  • 写入安全：precondition: {type: not_exists} 或 content_sha256 乐观并发

Hermes Agent：两文件 + 会话搜索

Hermes 的记忆系统最接近”笔记”的直觉：

文件 用途 上限
MEMORY.md Agent 的个人笔记（环境事实、约定、踩坑记录） 2,200 chars
USER.md 用户画像（偏好、沟通风格、身份） 1,375 chars

两者在会话启动时作为冻结快照注入系统提示，之后不再修改（工具响应中看到的是实时状态）。改变只在磁盘持久化，下一次会话才生效。

自进化机制：Agent 被” nudge”（轻推）提醒应该更新记忆——发生在对话中发现环境事实、约定或工具踩坑时。记忆超过 80% 容量时，nudge 会提示 consolidation。

会话搜索：所有对话存入 SQLite（~/.hermes/state.db）+ FTS5 全文索引，可搜索历史并用 Gemini Flash 做摘要总结。无需把全部历史塞进上下文。

外部记忆插件：Honcho、OpenViking、Mem0、Hindsight 等 8 种，可与内置记忆系统叠加使用。

OpenClaw：渐进式记忆注入

OpenClaw 的记忆更接近”档案室”：

  • MEMORY.md：长期事实和偏好，在每个 DM 会话开始时加载
  • memory/YYYY-MM-DD.md：每日append-only 日志，按需通过 memory 工具访问，不自动注入
  • DREAMS.md（实验性）：梦境日记， consolidation summaries 由 cron 定时运行，供人工审查
  • 自动记忆 flush：Compaction 之前会无声地提醒 Agent 把重要上下文写入记忆文件

记忆系统对比小结

维度 Claude MA Hermes OpenClaw
持久记忆 8 个 Memory Stores / 会话 MEMORY.md + USER.md MEMORY.md + memory/
总容量 ~25K tokens / 条 ~1.3K tokens 总计 按需加载
搜索 memory_search FTS5 + LLM 摘要 混合搜索（向量 + 关键词）
自进化 显式 memory_write 定期 nudge 触发 手动 / dreaming 实验
外部插件 — 8 种（Honcho 等） —

三、技能系统：按需加载的多种实现

三个工具都使用 SKILL.md + YAML frontmatter 格式，但实现细节差异很大。

Claude Managed Agents：Attached Skills

  • 技能在 Agent 创建时通过 skill_id 附加，最多 20 个
  • 技能文件系统化，加载时只影响相关上下文（不相关不注入）
  • Anthropic 提供预建技能：xlsx、pdf、word、powerpoint
  • 不支持自动创建或条件激活

Hermes 的技能系统最精细，三级加载：

条件激活（这是 Hermes 独有的）：

自动创建：当 Agent 完成复杂任务（5+ 工具调用）、找到错误修复路径、或被用户纠正时，会自动调用 skill_manage 工具创建技能。创建后跨会话持久化。

OpenClaw：紧凑列表 + 加载时过滤

OpenClaw 不把技能全文注入每个提示，而是在上下文组装阶段注入紧凑 XML 列表（name + description + file path）。模型只有在决定需要某个技能时才用 read 工具按需读取 SKILL.md。

  • 每个技能约 97 chars 开销，基础开销（>=1 技能时）约 195 chars
  • 技能可声明加载条件：requires.bins、requires.env、requires.config、os 平台限制
  • 支持 per-agent allowlist 和 deny

技能市场

工具 市场
Claude MA Anthropic 官方技能
Hermes 官方 + ClawHub + LobeHub + GitHub 官方技能 + Mintlify docs skills
OpenClaw ClawHub

安全提示：第三方技能可能包含恶意代码（提示注入、凭据盗取）。Cisco 研究员和 Snyk 安全审计在社区技能中发现过有意义的比例——安装前务必审查。

四、平台接入：聊天生态的覆盖差异

Claude Managed Agents：API 优先

不直接接入聊天平台，而是通过事件流（SSE + WebSocket）让你的应用接收 Claude 的响应、工具调用确认、状态变化等事件。

  • 通过 user.tool_confirmation 处理 always_ask 工具
  • 通过 user.interrupt 打断正在进行的任务
  • 多用户通过多 vault / 多 session 管理

Hermes Agent：15 种平台适配器

支持最多的聊天平台，每种都通过 Bot token + pairing 接入：

平台 语音 图片 文件 线程 流式
Telegram ✅ ✅ ✅ ✅ ✅
Discord ✅ ✅ ✅ ✅ ✅
Slack ✅ ✅ ✅ ✅ ✅
WhatsApp — ✅ ✅ — ✅
Signal — ✅ ✅ — ✅
SMS — — — — —
Email — ✅ ✅ ✅ —

安全模型：默认 DENY ALL（不在白名单或未配对的用户全部拒绝）。配对码 1 小时过期，有速率限制，密码学随机生成。

OpenClaw：Gateway 原生适配

Gateway 直接持有各平台会话，同样通过 Bot token 或官方协议接入：

  • 官方：Discord、Slack、Telegram、WhatsApp、Signal、Matrix、Microsoft Teams、iMessage 等
  • Bundled 插件：Matrix 扩展、Nostr、Twitch 等
  • 跨频道路由：不同 Agent 可以按频道、联系人或群组配置

五、安全模型：从凭证管理到权限控制

Claude Managed Agents：Vault 隔离

凭证永远不到达沙箱，是三者中最严格的：

  • Agent 定义只声明 MCP 服务器名称和 URL
  • Session 创建时传入 vault_ids
  • MCP proxy 在运行时根据 URL 匹配 vault 中的凭证并注入
  • 凭证分为 mcp_oauth（支持自动刷新）和 static bearer 两种
  • Secret 字段（token、refresh_token、client_secret）只在写入时返回，写入后不可读取

提示注入防御：结构化隔离——Git 的 access token 在沙箱初始化时绑定到 git remote，Agent 永远不接触 token；自定义 MCP 工具通过 dedicated proxy 访问 vault，harness 本身对凭证无感知。

权限策略：每个工具集可设置为 always_allow（自动执行）或 always_ask（暂停等待 user.tool_confirmation）。MCP 工具集默认为 always_ask。

Hermes Agent：环境变量 + 记忆扫描

  • API Key 通过环境变量或 ~/.hermes/env 配置，不在代码或配置文件中明文存储
  • 记忆注入扫描：检测 MEMORY.md 中是否包含提示注入、凭据外泄、SSH 后门等威胁模式，发现即拦截

OpenClaw：沙箱 + 绑定挂载防护

  • Docker 容器提供隔离环境，可选开启沙箱
  • 危险的绑定挂载源（docker.sock、/etc、/proc、/sys）和敏感凭据目录（~/.aws、~/.ssh、~/.gnupg）被主动阻止
  • Symlink-parent 逃脱是主要威胁向量
  • Safety guardrail 在系统提示中仅为建议性保护，强制执行需配合工具策略和 exec 审批

七、选型建议

场景 推荐
企业 / 生产环境，长期任务 Claude Managed Agents（官方托管、Vault 安全、多 Agent 编排、Memory Stores）
个人 / 开源爱好者 / 自进化需求 Hermes Agent（自动建技能、18+ 模型、FTS5 会话搜索、多 Profile 隔离）
接聊天平台 / 个人助手 / 多频道 OpenClaw（Gateway 架构、15+ 平台、ClawHub 生态、零成本入门）

三条路线的共同趋势：都在用接口抽象代替硬编码，让 Agent 的各个组件可以独立演进。Anthropic 学的是 OS，Hermes 做的是单引擎多入口，OpenClaw 则把网关变成了可组合的插件系统。
